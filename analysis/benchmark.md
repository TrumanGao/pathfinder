# Empirical Benchmarks

This document reports measurements from the `BenchmarkTool` harness on
the real DC-area road network (1.88 M nodes, 3.96 M directed edges),
and compares them to the asymptotic predictions in [complexity.md](complexity.md).

Raw CSV data for every run lives in [results/](results/).

## Methodology

### Routing comparison

- **Dataset**: `data/full.geojson` loaded via the binary cache
  (post-warmup, so graph I/O is out of the hot path).
- **Pair selection**: 50 random (start, end) pairs, seeded with `42`
  for reproducibility. Each endpoint is drawn uniformly from nodes
  with at least one outgoing edge.
- **Cost model**: pure distance
  (`PathCostModel.distanceOnly()`) — so the three algorithms are
  comparing like with like and cross-check on distance equality is
  meaningful.
- **Per-algorithm instrumentation**: the real `PathCostModel` is
  wrapped in a `CountingCostModel` that increments an `AtomicLong`
  every time `edgeCost(...)` is called. This count is the total
  **edge relaxations**, which is the fundamental work unit of
  Dijkstra-family algorithms — independent of JVM warmup, GC, or CPU
  frequency scaling.
- **Timing**: `System.nanoTime()` around the top-level `findPath`
  call, converted to milliseconds. Reverse-adjacency index is pre-warmed
  before the first bidirectional query so no one-time build cost leaks
  into the measurements.

### Nearest-neighbour comparison

- **Queries**: 50 synthetic query points generated by jittering a
  random node's coordinates by up to ~100 m, so the query never
  trivially lands on an existing node.
- **Linear scan** sweeps all 1.88 M nodes per query, computing a full
  haversine every time.
- **kd-tree** uses the production service's tree.

## Results

### Routing algorithms

```
algorithm      pairs  avg_edge_relaxations  avg_time_ms  median_time_ms  correctness
dijkstra       50     2,164,413             1150.0       1211.5          48/48 match (self)
astar          50     538,737               334.5        270.8           48/48 match vs Dijkstra
bidirectional  50     1,487,568             962.3        952.9           48/48 match vs Dijkstra
```

**Correctness.** Two of the fifty pairs turn out to be unreachable
(disconnected components in the OSM data). The remaining 48 pairs all
produce **identical total distances** across all three algorithms to
numerical precision (1e-6 m tolerance), which is the empirical proof
that our Bi-Dijkstra implementation is correct and that A* never
over-explores.

**Performance vs Dijkstra.**

| Algorithm | Relaxation reduction | Wall-time speedup |
|---|---|---|
| A* | **4.0×** fewer | **3.4×** faster |
| Bi-Dijkstra | 1.5× fewer | 1.2× faster |

**Observations.**

1. **A\* dominates on this dataset.** Haversine is a remarkably tight
   heuristic on road networks — most roads follow the underlying
   geography, so the lower bound `H(v, t)` is very close to the actual
   shortest path cost. The relaxation count drops to roughly 25 % of
   Dijkstra's.

2. **Bi-Dijkstra's advantage is modest but real.** 1.5× fewer
   relaxations matches the textbook expectation of "roughly two balls
   of radius *d*/2 vs one ball of radius *d*", which is a factor
   ~2 in 2D but less at real road-network geometry (the balls are not
   perfectly circular and the road graph has anisotropic connectivity).

3. **Wall time / relaxation ratio differs across algorithms.** Dijkstra
   spends ~531 ns per relaxation on average, A* ~621 ns (heuristic
   computation is extra work per relaxation), Bi-Dijkstra ~647 ns (two
   heaps, more bookkeeping). This is where A\*'s per-relaxation
   overhead almost catches up to its relaxation savings on shorter
   queries, but wins handily on longer ones.

### The unreachable-pair case — a striking win for Bi-Dijkstra

Pair #2 (`n:38.9350257,-77.2647553` → `n:39.0178306,-77.0656611`) is
disconnected in the graph. Dijkstra and A\* cannot prove non-existence
without exhausting the entire reachable component from the start —
essentially `O(E)` work:

| Algorithm | Relaxations | Time |
|---|---|---|
| Dijkstra | 3,931,698 (≈ total edges) | 2,245 ms |
| A* | 3,931,698 | 2,524 ms |
| Bi-Dijkstra | **64** | **0.17 ms** |

Both frontiers of Bi-Dijkstra exhaust their respective connected
components after a handful of steps and return "not found" immediately.
This is a **12,000× speedup** on the pathological case, and a real
argument for using Bi-Dijkstra as the algorithm of record for any
workload that might include unreachable pairs.

### Nearest-neighbour

```
method          queries   avg_us_per_query   speedup_vs_linear
linear_scan     50        87,616.9           1.00×
kd_tree         50        39.7               2,204.97×
```

**2,200× faster.** A single click on the map used to cost ~88 ms of
pure CPU, which is noticeable in user latency on top of frontend work;
now it costs ~40 microseconds. This aligns with the theoretical
`O(V)` → `O(log₂ V) ≈ O(21)` prediction (1.88 M / 21 ≈ 90,000, close
to the observed ratio, the rest being haversine computation overhead
amortised across fewer visits).

**Build cost.** 565 ms over 1.88 M nodes, amortised across every
nearest-neighbour query for the process lifetime. Even with a single
query per run, the break-even point is ~6 queries.

### Graph cache

```
first run (parse + write):   9.65 s parse + 4.11 s write = 13.76 s total graph load
subsequent runs (read):      1.85 s

reduction: 7.45×
```

Full Spring Boot startup correspondingly drops from **~19.5 s** to
**~4.4 s** (cache-hit path). The 279 MB cache file stores the same
graph that the GeoJSON parse would produce, at ~30 % of the raw
GeoJSON size, with all per-string dedup work pre-computed via the
string table.

## What the numbers confirm (and don't)

### Confirmed
- **Adjacency list is the right choice**: every routing query touches
  at most ~4 M edge slots, far below the ~3.5 × 10¹² cells that a
  dense matrix would require.
- **A\* on Haversine is near-optimal** for this dataset. Unless the
  cost model drifts far from distance (e.g. heavy safety/time blending
  that breaks the heuristic's tightness), A\* should remain the default.
- **kd-tree delivers the textbook log-factor speedup** on nearest
  neighbour. The constant factor is excellent (~40 µs/query).
- **Bi-Dijkstra's killer use case is unreachable pairs**, not average
  reachable pairs.

### Not (yet) confirmed
- **Bi-A\* with symmetric potentials** should beat plain A\* on long
  queries. We did not implement this because the correctness
  argument is non-trivial (see [complexity.md § Bi-A\*](complexity.md#why-not-bidirectional-a)).
  Not running the experiment is itself a choice: implementation risk
  outweighs a plausible 2× speedup on already-fast A\* queries.
- **Contraction hierarchies / ALT** would likely reduce per-query
  time by another 1-2 orders of magnitude by pre-processing the graph
  further. These are out of scope for a course project but worth
  mentioning as the next rung on the ladder.

## How to reproduce

```bash
cd server
./mvnw.cmd org.codehaus.mojo:exec-maven-plugin:3.1.0:java \
    -Dexec.classpathScope=test \
    -Dexec.mainClass=edu.northeastern.pathfinder.benchmark.BenchmarkTool \
    -Dexec.args="--pairs 50 --seed 42 --out ../analysis/results/benchmark-latest.csv"
```

Adjust `--pairs` and `--seed` for different sample sizes and random
seeds. All runs in this document use `--pairs 50 --seed 42`.

`JAVA_HOME` must point to a JDK 17+ installation — the project does
not build on Java 8, and the system default may be older.
